## Detailed Fixes

### 1. AvailabilityManager.tsx - Invalid `title` Prop on Lucide Icon

**Error:**
```
Property 'title' does not exist on type 'IntrinsicAttributes & LucideProps'
```

**Root Cause:** The `title` attribute was being passed directly to a Lucide React icon component (`AlertCircle`), but Lucide icons don't accept `title` as a prop.

**Fix:** Changed `title` to `aria-label` for proper accessibility support:

```tsx
// Before
<span title={slotError}>
  <AlertCircle className="h-4 w-4 text-yellow-500" />
</span>

// After
<span aria-label={slotError}>
  <AlertCircle className="h-4 w-4 text-yellow-500" />
</span>
```

### 2. NotificationBell.tsx - Missing Socket Service Method

**Error:**
```
Property 'off' does not exist on type 'SocketService'
```

**Root Cause:** The code was trying to use `socketService.off('notification:new', handleNewNotification)` to unsubscribe, but the SocketService didn't have an `off` method implemented in this version.

**Fix:**

2. Updated cleanup in NotificationBell to use the new pattern

```tsx

  return this.on('notification:new', callback);
}

onBookingStatusChanged(callback: (data: BookingEvent) => void): () => void {
  return this.on('booking:status_changed', callback);
}

off(event: keyof ServerToClientEvents, callback: ServerToClientEvents[keyof ServerToClientEvents]): void {
  this.listeners.get(event)?.delete(callback as Function);
}

// NotificationBell.tsx - Updated cleanup:
return () => {
  const unsubscribe = socketService.onNewNotification(handleNewNotification);
  unsubscribe();
  socketService.disconnect();
};
```

### 3. ServiceManagement.tsx - StatusCounts Type Mismatch

**Error:**
```
Argument of type 'ServiceStats | StatusCounts' is not assignable to parameter of type 'SetStateAction<StatusCounts>'
```

**Root Cause:** The API could return either `statusCounts` or `serviceStats`, but these have different shapes. `ServiceStats` has `total` while `StatusCounts` has `all`.

**Fix:** Restructured to handle both cases properly:

```tsx
// Before - assumed both types had the same properties
const statusData = data.data.overview.statusCounts || data.data.overview.serviceStats;
setStatusCounts({
  all: statusData.all ?? statusData.total ?? 0,
  // ...
});

// After - check which type we received
if (statusCounts) {
  setStatusCounts(statusCounts);
} else {
  const stats = data.data.overview.serviceStats;
  setStatusCounts({
    all: stats.total ?? 0,
    active: stats.active ?? 0,
    inactive: stats.inactive ?? 0,
    pending_review: stats.pending_review ?? 0,
    draft: stats.draft ?? 0,
  });
}
```

### 9. NotificationAnalyticsService.ts - Backend Response Format Mismatch

**Error:**
```
Type '{ totalSent: number; totalDelivered: number; ... }' is missing properties 'channels' and 'recentActivity'
```

**Root Cause:** The backend API returns analytics in a different format (`byChannel`, `byType`) than what the service expects (`channels`, `recentActivity`).

**Fix:** Added an adapter function to transform the backend response:

```tsx
async getAnalyticsFromBackend(params?: {...}): Promise<NotificationAnalyticsData | null> {
  const response = await notificationApi.getAnalytics(params);
  if (response.success && response.data) {
    // Adapt backend response format to NotificationAnalyticsData
    const adaptedData: NotificationAnalyticsData = {
      channels: {
        in_app: { sent: 0, delivered: 0, clicked: 0, rate: 0 },
        email: { sent: 0, delivered: 0, clicked: 0, rate: 0 },
        sms: { sent: 0, delivered: 0, clicked: 0, rate: 0 },
        push: { sent: 0, delivered: 0, clicked: 0, rate: 0 },
      },
      recentActivity: { last7Days: 0, last30Days: 0 },
    };
    // Map byChannel to channels
    Object.entries(response.data.byChannel).forEach(([channel, stats]) => {
      if (channel in adaptedData.channels) {
        adaptedData.channels[channel as keyof typeof adaptedData.channels] = {
          sent: stats.sent,
          delivered: stats.delivered,
          clicked: stats.clicked,
          rate: stats.rate,
        };
      }
    });
    // ... merge with local analytics
  }
}
```

## Notes for Future Reference

1. **Toast API**: The codebase uses `addToast({ title, variant })` pattern. Avoid using `toast.error()` or `toast.success()` directly.

2. **SocketService**: When unsubscribing, use the unsubscribe function returned by `onNewNotification()`, `onBookingStatusChanged()`, etc. Avoid direct `off()` calls unless necessary.

3. **Type Safety**: When casting between unrelated types, always use `as unknown as TargetType` pattern.

4. **API Response Shapes**: Backend and frontend type definitions can drift. Always check the actual API response shape before assuming properties exist.
