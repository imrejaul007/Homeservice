## 1. Recommended Pros Section - Distance Fix

### Problem
The "Recommended Pros" section on the dashboard wasn't showing distance for providers even though geolocation data existed in the database.

### Root Causes Identified

1. **Controller not passing coordinates** - `customerDashboard.controller.ts` wasn't passing latitude/longitude to the service
2. **Wrong field path in query** - Service was querying `providerId.location.coordinates` instead of `locationInfo.primaryAddress.coordinates`
3. **Missing field mappings** - Service responses weren't mapping fields correctly for frontend consumption

### Files Fixed

#### `backend/src/controllers/customerDashboard.controller.ts`
```typescript
// BEFORE: Coordinates not passed to service
const pros = await customerDashboardService.getRecommendedPros(
  userId,
  tenantId,
  value.limit
);

// AFTER: Coordinates passed correctly
const pros = await customerDashboardService.getRecommendedPros(
  userId,
  tenantId,
  value.limit,
  {
    latitude: value.latitude,
    longitude: value.longitude,
    maxDistanceKm: value.radius,
  }
);
```

#### `backend/src/services/customerDashboard.service.ts`
```typescript
// BEFORE: Wrong field path
'providerId.location.coordinates': { $exists: true, $ne: null }

// AFTER: Correct field path
'locationInfo.primaryAddress.coordinates': { $exists: true, $ne: null }
```

Also fixed service field mapping in 3 places (getServicePackages, getPackageById, getFeaturedPackages):
```typescript
services: (bundle.services || []).map((s: any) => ({
  _id: s.serviceId,
  name: s.serviceName,
  serviceName: s.serviceName,
  duration: s.duration || 60,
  price: s.originalPrice || s.price || 0,
  originalPrice: s.originalPrice,
  description: s.description,
})),
```

## 9. Price Conversion Utility (Added 2:30 PM)

### Purpose
Convert prices from USD (database) to local currency based on user's selected location in the header.

### New Files Created

#### `frontend/src/utils/priceConverter.ts`
```typescript
// Key exports:
- usePriceConversion() // React hook
- convert(price, fromCurrency, toCurrency)
- format(price, currency)
- getCurrencyFromCity(cityName)
```

#### `frontend/src/components/common/PriceDisplay.tsx`
- `PriceDisplay` - basic price display component
- `ServicePriceDisplay` - with savings calculation

### Supported Currencies & Exchange Rates
| Location | Currency | Rate (from USD) |
|----------|----------|------------------|
| Dubai/UAE | AED | 3.67 |
| Saudi Arabia | SAR | 3.75 |
| UK | GBP | 0.79 |
| Europe | EUR | 0.92 |
| India | INR | 83.12 |
| Pakistan | PKR | 278.5 |

### How It Works
1. User changes location in header → `useLocationStore.setSelectedCity()`
2. Store triggers re-render in all components
3. `usePriceConversion()` hook reads `selectedCity`
4. `getCurrencyFromCity()` maps city to currency
5. All prices display via `format(convert(price, 'USD'), currency)`

### Example Conversions
- $100 USD in Dubai → "AED 367"
- $100 USD in India → "₹8,312"
- $100 USD in Pakistan → "Rs 27,850"

## 21. Chat Message Infinite Loop Fix — Active Bookings → Messages (Added)

### Purpose
Fix React "Maximum update depth exceeded" / `useEffect` infinite loop when opening a conversation from **Active & Today's Bookings** via the **Message** button. The loop flooded the browser with chat API calls (`GET /messages`, `PATCH /read`) until `ERR_INSUFFICIENT_RESOURCES`.

### User Report
> Clicking Message on a booking card caused `useEffect` dependency churn, repeated network errors, and chat failing to open.

### End-to-End Flow (Before Fix)

```
OngoingBookings (Message button)
  → navigate('/customer/messages', { providerId, bookingId })
MessagesPage (deep-link effect)
  → chatApi.getOrCreateBookingChat()
  → setSelectedRoom(chatRoom)
ChatWindow
  → initialSelectedRoom={selectedRoom}
  → onSelectRoom={setSelectedRoom}   ← feedback loop
```

| Step | Component | Action |
|------|-----------|--------|
| 1 | `OngoingBookings.tsx` | `handleMessageProvider` → navigate with `providerId` + `bookingId` |
| 2 | `MessagesPage.tsx` | Deep-link `useEffect` creates/gets booking chat room, sets `selectedRoom` |
| 3 | `ChatWindow.tsx` | `initialSelectedRoom` effect called `handleSelectRoom` |
| 4 | `ChatWindow.tsx` | `handleSelectRoom` called `onSelectRoom` → parent `setSelectedRoom` |
| 5 | Parent re-render | New object reference for same room → effect re-ran → infinite fetches |

### Root Cause

1. **Feedback loop** — `initialSelectedRoom` effect depended on `handleSelectRoom`, which called `onSelectRoom(room)`. Parent updated state with a new object (via `normalizeChatRoom`) even when room ID was unchanged → effect re-triggered endlessly.
2. **Duplicate fetch paths** — `handleSelectRoom` called `fetchMessages` while a broken mount `useEffect` (`[]` deps) also tried to load messages with stale `selectedRoom`.
3. **Back button gap** — Back only cleared parent `selectedRoom`; local `selectedRoomState` in `ChatWindow` stayed set.
4. **Socket/mount effect** — Single empty-deps effect mixed room fetch, message fetch, and socket join with stale closures.

### What Was Fixed

#### `frontend/src/components/chat/ChatWindow.tsx` (MODIFIED)

**Room ID guards (refs)**
- `appliedInitialRoomIdRef` — tracks parent deep-link room already applied
- `activeRoomIdRef` — tracks currently open room for socket handlers
- `getRoomId()` helper — stable ID comparison across `_id` / `id`

**Broken loop removed**
- Initial-room sync effect: apply room **by ID once**; do **not** call `onSelectRoom` (parent already has the room)
- `handleSelectRoom`: sets local state + notifies parent; no direct `fetchMessages` call

**Effects split (single responsibility)**
| Effect | Trigger | Action |
|--------|---------|--------|
| Mount | once | `fetchChatRooms` |
| `selectedRoomId` | room ID change | `fetchMessages` + abort prior request |
| Socket | mount | subscribe to `message:new`, typing, delivered (uses `activeRoomIdRef`) |
| `selectedRoomId` | room ID change | `joinChatRoom` / `leaveChatRoom` |

**Back navigation**
- `handleBackToRoomsInternal` clears refs, local room, messages, typing state, then calls parent `onBackToRooms`

**Callback order**
- Moved `markMessagesAsRead` above `fetchMessages` so dependency array is valid

### Also Benefits

Same fix applies to `ChatWidget.tsx`, which uses the same `initialSelectedRoom` + `onSelectRoom={setSelectedRoom}` pattern.

### Files Modified

| File | Change |
|------|--------|
| `frontend/src/components/chat/ChatWindow.tsx` | Break infinite loop; split effects; room ID guards; fix back navigation |

### Related Files (unchanged, flow verified)

| File | Role |
|------|------|
